# Планирование фич

## Overview

План описывает **КАК СИСТЕМА БУДЕТ РАБОТАТЬ**, а не только что написать. Каждая фича проходит через 8 фаз анализа, прежде чем появятся задачи.

**Объявить при старте:** "Использую скилл planning-features для анализа полноты плана."

## Когда использовать

- Новая фича, затрагивающая >2 слоёв системы
- Интеграция с внешним API/сервисом
- Расширение или замена существующего поведения

## Когда НЕ использовать

- Однослойные изменения (фикс бага, добавление поля в endpoint)
- Рефакторинг без изменения поведения
- Изменение одного handler без побочных эффектов

## Процесс (8 фаз)

**ОБЯЗАТЕЛЬНО:** пройти все 8 фаз последовательно. Пропуск фазы = неполный план.

### Фаза 1: User Flow

Описать полный путь пользователя от начала до конца, **включая все ветки**.

Формат для каждого пути:
```
Актор → Действие → Результат
  → если ошибка → [описание обработки]
  → если повтор → [описание поведения]
```

Обязательные пути:
- **Happy path** — основной сценарий от начала до конца
- **Error paths** — ошибка API, невалидные данные, таймаут
- **Repeat paths** — F5 на странице, повторный вызов, retry после ошибки
- **Admin paths** — бот, админка, ручные операции

**Каждая ветка с ошибкой = минимум один тест.**

Задачи извлекаются ТОЛЬКО после полного описания flow.

### Фаза 2: Сравнение с существующим

Пропустить если нет аналога. Если аналог есть — составить таблицу:

| Шаг | Текущее поведение | Новое поведение | Отличие |
|-----|-------------------|-----------------|---------|

**Каждое непустое "Отличие" = отдельная задача или подзадача.**

Сравнивать шаг за шагом: создание, обработку, формат данных, аутентификацию, формат ошибок, конфигурацию, поддерживаемые методы (нужен ли новый enum value?).

### Фаза 3: Edge Cases по каждому handler

Для **КАЖДОГО** handler/endpoint из Фазы 1 задать **ВСЕ 7 вопросов**:

1. **Идемпотентность** — Что если вызвали повторно? (F5, retry, дубль)
2. **Несовпадение данных** — Что если данные в запросе не совпадают с БД?
3. **Nil зависимости** — Что если subscription/user/payment = nil?
4. **Заблокированные ресурсы** — Что если предыдущий шаг заблокировал ресурс? (скидки, лимиты, промокоды, балансы). Если шаг N+1 упал — нужен rollback шага N.
5. **Ошибка внешнего API** — Что если API вернул ошибку или таймаут?
6. **Null/пустые входные данные** — Что если опциональные поля = null? (не пустая строка, а null)
7. **Расхождение с документацией** — Что если реальный формат данных отличается от документации?

Формат — таблица per handler. Каждый ответ "нужна обработка" = задача.

### Фаза 4: Все точки входа

**Вопрос:** "Кто использует данные, которые я меняю?"

Обязательно проверить:
- [ ] HTTP endpoints (REST API)
- [ ] Webhook handlers (входящие)
- [ ] Workers / consumers (очереди, cron)
- [ ] Bot handlers (Telegram, Slack)
- [ ] Admin UI
- [ ] Status / health endpoints
- [ ] Sync workers (payment_sync, reconciliation)
- [ ] Stat / analytics сервисы

**Каждая точка входа, требующая изменений = задача в плане.**

### Фаза 5: Инфраструктурный чек-лист

При **ЛЮБОМ** изменении модели или контракта:

**Модель данных:**
- [ ] Новое поле в декларации модели → кодогенерация
- [ ] SQL-миграция для нового поля (`ALTER TABLE ... ADD COLUMN`)
- [ ] Обновление schema-файлов
- [ ] Новое значение в enum → миграция (`ALTER TYPE ... ADD VALUE`)
- [ ] Обновление файлов типов

**Контракты (проверить ВСЕ, не только основной):**
- [ ] Public API контракт → кодогенерация
- [ ] Stat/analytics контракт — нужен ли **новый endpoint и новая схема** (не просто поле)? → кодогенерация + мок
- [ ] Webhook контракт → nullable поля по реальным данным (не по документации!)
- [ ] Internal API контракт (gRPC/proto)

**Конфигурация:**
- [ ] Структура конфигурационных путей спроектирована
- [ ] Тестовая конфигурация обновлена
- [ ] Production конфигурация задокументирована

**Тулинг:**
- [ ] Версии генераторов совместимы с новым кодом
- [ ] `make regenerate` (или аналог) проходит чисто ДО начала работы
- [ ] Старый неиспользуемый код удалён (не мешает генерации)

### Фаза 6: Реальные данные

**НЕ писать контракт по документации провайдера.** Документация врёт, устарела, или описывает идеальный случай.

Шаги:
- [ ] Зарегистрировать sandbox-аккаунт
- [ ] Выполнить тестовый запрос
- [ ] Получить реальный response/webhook JSON
- [ ] Сравнить реальный JSON с документацией
- [ ] Выявить nullable поля (null, а не пустая строка)
- [ ] Выявить недокументированные поля
- [ ] Зафиксировать расхождения

**Если sandbox недоступен** — зафиксировать риск явно в плане. Контракт = предположение, которое БУДЕТ пересмотрено.

### Фаза 7: Проектирование ошибок ДО happy path

Для каждого шага из Фазы 1 заполнить:

| Шаг | Заблокированные ресурсы | Rollback при ошибке | Ошибка пользователю | Retry стратегия |
|-----|------------------------|--------------------|--------------------|----------------|

**Ключевой вопрос:** "Если на шаге N заблокирован ресурс, а на шаге N+1 ошибка — что откатывать?"

Это **архитектурное решение**, а не "поправка по ходу". Проектировать ДО happy path.

Для fallback-систем: описать **и клиентский, и серверный** fallback. Они работают по-разному:
- Клиентский: клиент отправляет `fallback=true` → сервер пробует следующего
- Серверный: провайдер вернул ошибку → сервер автоматически пробует следующего
- Это **разные механизмы** с разной архитектурой.

### Фаза 8: Проверка по всем слоям

Финальная проверка — для фичи пройти по каждому слою:

| Слой | Нужны изменения? | Задача |
|------|:----------------:|--------|
| Контракт (OpenAPI/proto/swagger) | | |
| БД (миграция + модель + schema) | | |
| Сервис (бизнес-логика) | | |
| Handler (HTTP/gRPC endpoints) | | |
| Webhook handler (входящие) | | |
| Stats / analytics | | |
| Конфигурация (OnlineConf/env) | | |
| Admin UI / Bot | | |
| Тесты (unit + integration + e2e) | | |
| Тулинг (codegen, генераторы) | | |

Непроверенный слой = потенциально пропущенная задача.

## Чек-лист перед утверждением плана

**Не утверждать план, пока все пункты не пройдены:**

- [ ] **Flow описан полностью** — от действия пользователя до результата, включая ошибки и повторы
- [ ] **Все точки входа перечислены** — API, bot, workers, status, sync
- [ ] **Сравнение с аналогом сделано** — таблица отличий, каждое отличие = задача
- [ ] **Реальные данные получены** — sandbox response/webhook проверены (или риск зафиксирован)
- [ ] **Миграции БД включены** — каждое поле = миграция + schema + codegen
- [ ] **ВСЕ контракты обновлены** — public API + stat + webhook + internal
- [ ] **Конфигурация спроектирована** — структура путей, тестовые значения
- [ ] **Тулинг проверен** — генерация проходит чисто до начала работы
- [ ] **Edge cases описаны** — для каждого handler все 7 вопросов
- [ ] **Тесты названы конкретно** — TestPaymentTokenFallback, не "написать тесты"
- [ ] **Error handling спроектирован** — для каждого шага: что откатывать при ошибке
- [ ] **Fallback стратегия полная** — и клиентский, и серверный fallback

## Типичные ошибки

| Ошибка | Реальность |
|--------|-----------|
| "Сделаю как для X" | У каждого аналога свой flow. Нужна таблица сравнения |
| "Тесты напишу потом" | Без имён тестов в плане edge cases будут пропущены |
| "Миграция — это очевидно" | Каждое поле = декларация + migration SQL + schema SQL |
| "Контракт по документации" | Документация врёт. Нужны реальные данные |
| "Обработку ошибок добавлю по ходу" | Error handling = архитектурное решение, не поправка |
| "Это только изменение API" | Любое изменение API затрагивает ВСЕ слои |
| "Fallback = просто retry" | Client fallback и server auto-fallback — разные механизмы |
| "Stat-контракт обновлю потом" | Забудешь. Включай в план сразу |

## Red Flags — СТОП, план неполный

Если в плане есть хотя бы один из этих признаков — вернуться к соответствующей фазе:

- Описан только happy path → вернуться к Фазе 1
- Нет edge cases по handler → вернуться к Фазе 3
- Миграции БД не упомянуты → вернуться к Фазе 5
- Обновлён только один контракт (их всегда несколько) → вернуться к Фазе 5
- "Добавлю обработку ошибок потом" → вернуться к Фазе 7
- "Аналогично существующему X" без таблицы → вернуться к Фазе 2
- Тесты без конкретных имён → переименовать

## Интеграция

```
brainstorming → [planning-features] → writing-plans → executing-plans
```

- **Вход:** дизайн-документ из brainstorming или описание фичи от пользователя
- **Выход:** полный план с задачами, покрывающий все слои и edge cases
- **После:** `superpowers:writing-plans` для детализации задач в TDD-шаги
