# Add GOAT Tests to Existing Project

Interactive command to add GOAT (Go Async Testing) integration tests to an existing go-project-starter project.

## Prerequisites

**CRITICAL: `main.name` must NOT contain hyphens.** The generator uses `main.name` to create Go type names (e.g., `MyserviceConfig`). Hyphens are invalid in Go identifiers — a name like `my-service` generates `My-ServiceConfig` which won't compile. If `main.name` has hyphens, rename it first (e.g., `myservice` or `myService`), then run `make regenerate` before adding GOAT tests. This rename affects OnlineConf env var prefixes (`OC_{name}__...`), `.env` files, and `constant.ServiceName`.

## Mandatory Steps

The task is NOT complete until ALL of these pass:
1. Read existing project.yaml
2. **Verify `main.name` has no hyphens** (fix if needed)
3. Interview the user if needed
4. Modify project.yaml
5. Run `make regenerate` successfully
6. Run `make build` successfully (no compilation errors)

## Step 1: Read Existing Config

```bash
cat .project-config/project.yaml
```

Identify:
- Which applications exist
- Which application to add tests to (usually the one with REST API)
- Whether goat_tests is already enabled

## Step 2: Interview

Ask the user:

1. **Which application?** — which app should have GOAT tests (typically the main API app)
2. **Binary path** — test binary location (default: `/tmp/{app_name}`)

## Step 3: Modify Config

Add `goat_tests` or `goat_tests_config` to the target application:

### Simple flag:
```yaml
applications:
  - name: api
    goat_tests: true              # Simple: uses default binary_path /tmp/{app_name}
    transport:
      - name: api
      - name: sys
```

### Extended config:
```yaml
applications:
  - name: api
    goat_tests_config:
      enabled: true
      binary_path: "/tmp/api"     # Custom binary path
    transport:
      - name: api
      - name: sys
```

## Step 4: Regenerate

```bash
make regenerate
```

This generates:
- `tests/{app_name}/psg_main_test.go` — TestMain entry point
- `tests/{app_name}/psg_base_suite_gen.go` — BaseTestSuite with setup/teardown
- `tests/{app_name}/psg_app_gen.go` — test app configuration
- `tests/{app_name}/psg_config_gen.go` — test config
- `tests/{app_name}/psg_helpers_gen.go` — test helpers

## Step 5: Build

```bash
make build
```

Also verify test infrastructure compiles:
```bash
make build-for-test-{app_name} 2>&1 || true  # May need test env setup
```

## Step 6: Summary

Tell the user what was generated and how to use it:

### Writing Tests

Create test files in `tests/{app_name}/`:

```go
package tests

import (
    "testing"
    "github.com/stretchr/testify/suite"
)

type MyTestSuite struct {
    BaseTestSuite    // Generated: provides ctx, HTTP clients, mocks
}

func TestMyTestSuite(t *testing.T) {
    suite.Run(t, new(MyTestSuite))
}

func (s *MyTestSuite) TestHealthCheck() {
    resp := s.sysClient.Get("/ready")
    s.Equal(200, resp.StatusCode)
}
```

### Key Rules

- **NEVER call service functions directly** from GOAT tests. The test context doesn't have OnlineConf properly initialized. Always test through API endpoints (HTTP clients).
- Available HTTP clients: one per REST transport (e.g., `s.apiClient`, `s.sysClient`)
- `SetupSuite()` initializes DB, ActiveRecord, HTTP mocks
- `TearDownTest()` cleans test data between tests

### Running Tests

```bash
make goat-tests-{app_name}                    # Run all GOAT tests
make goat-tests-{app_name}-run TEST=TestSuite/TestMethod  # Run specific test
make build-for-test                            # Rebuild test binary after code changes
```

### Test Configuration

- Test OnlineConf: `tests/etc/onlineconf/TREE.conf` (editable)
- Production OnlineConf: `etc/onlineconf/` (NEVER edit)
- Test init: `tests/{app_name}/init.go` — customize test setup here
